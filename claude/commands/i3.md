# /i3 - i3 Window Manager Interaction

## Triggers
- Requests to query or manipulate windows, workspaces, or layouts
- Questions about what's running on which workspace
- Requests to arrange windows or set up multi-window layouts
- Desktop notification requests
- Window focus or movement commands
- Requests to type into, click on, or interact with GUI applications
- Requests to see/screenshot what's on screen
- Clipboard read/write operations

## Usage
```
/i3 <action> [args...]

Window Management:
  status          Show current workspace, focused window, and layout overview
  tree [filter]   Show container tree (filter: workspace num, window class, or "focused")
  workspaces      List all workspaces with window counts and focused state
  windows [ws]    List all windows (optionally filtered to workspace number)
  focus <target>  Focus a window by class, title substring, mark, or con_id
  move <target> <workspace>  Move a window to a workspace
  workspace <n>   Switch to workspace number/name
  layout <type>   Change focused container layout (split/tabbed/stacking)
  exec <cmd>      Launch an application
  mark <name>     Mark the focused window
  goto <mark>     Focus marked window
  resize <dir> <px>  Resize focused window
  scratch [show|hide|toggle]  Scratchpad operations
  config          Show current i3 config and keybindings summary
  find <query>    Find windows by class/title/instance (criteria search)
  arrange <recipe>  Apply a layout recipe (see recipes below)
  notify <msg>    Send desktop notification via dunst

Keyboard & Mouse:
  screenshot [window_class]  Capture full screen or specific window, display for analysis
  observe         Screenshot + analyze what's visible + suggest next actions
  type <text>     Type text into the focused window via xdotool
  key <combo>     Send key combination (e.g., ctrl+s, Return, alt+F4, super+l)
  click <x> <y>   Left-click at absolute screen coordinates
  click-on <class> <x> <y>  Click relative to a window's top-left corner
  rclick <x> <y>  Right-click at screen coordinates
  scroll <up|down> [n]  Scroll wheel in focused window (default: 3 clicks)
  drag <x1> <y1> <x2> <y2>  Click-drag from one point to another
  clipboard [get|set <text>]  Read or write the X11 clipboard
  mouse           Report current mouse position and window under cursor

Recipes (for `arrange`):
  dev             Terminal left 60%, browser right 40%
  monitor         Split into 3 columns for dashboards
  pair            Two terminals side by side 50/50
  present         Single fullscreen window
```

## Behavioral Flow

### 1. Query Phase
For read operations (`status`, `tree`, `workspaces`, `windows`, `find`):
- Use `i3-msg -t get_tree` to get full container tree
- Parse JSON with `jq` to extract relevant info
- Present in clean, readable format

**IMPORTANT: zsh/Bash tool escapes `!` in all strings. Never use `!=` in jq filters.**
**Use alternatives: `| not`, `| type == "number"`, `| . > 0`, `select(X) | select(Y)` instead of `select(X and Y)`.**

**Status query pattern:**
```bash
# Workspaces with focused indicator
i3-msg -t get_workspaces | jq -r '.[] | "\(if .focused then ">" else " " end) [\(.name)] on \(.output)\(if .urgent then " URGENT" else "" end)"'

# Focused window (use double-select to avoid "and ... != null")
i3-msg -t get_tree | jq '.. | select(.focused? == true) | select(.window? | type == "number") | {class: .window_properties.class, title: .name, id: .id}'

# All windows per workspace (use "startswith | not" instead of "!=")
i3-msg -t get_tree | jq '[recurse(.nodes[]?, .floating_nodes[]?) | select(.type == "workspace") | select(.name | startswith("__") | not) | {workspace: .name, windows: [recurse(.nodes[]?, .floating_nodes[]?) | select(.window | type == "number") | {class: .window_properties.class, title: (.name | if length > 60 then .[:57] + "..." else . end), focused: .focused}]}]'
```

### 2. Action Phase
For write operations (`focus`, `move`, `workspace`, `layout`, `exec`, `mark`, `resize`, `scratch`):
- Validate target exists before acting (query tree first for focus/move)
- Use criteria selectors when targeting by class/title: `i3-msg '[class="Firefox"] focus'`
- Use `i3-msg <command>` for direct commands
- Report result (i3-msg returns `[{"success": true/false}]`)

**Common action patterns:**
```bash
# Focus by class
i3-msg '[class="Brave-browser"] focus'

# Focus by title substring
i3-msg '[title=".*pattern.*"] focus'

# Move focused window to workspace
i3-msg 'move container to workspace number 3'

# Move specific window to workspace
i3-msg '[class="Brave-browser"] move container to workspace number 2'

# Change layout
i3-msg 'layout tabbed'
i3-msg 'layout toggle split'

# Resize
i3-msg 'resize grow width 100 px'
i3-msg 'resize set 60 ppt 0 ppt'

# Scratchpad
i3-msg 'move scratchpad'
i3-msg 'scratchpad show'

# Mark and goto
i3-msg 'mark mymark'
i3-msg '[con_mark="mymark"] focus'

# Launch app (always use --no-startup-id)
i3-msg 'exec --no-startup-id brave'
```

### 3. Compound Operations
For `arrange` recipes:
- Query current state to understand starting point
- Execute a sequence of i3 commands (move, resize, layout, focus)
- Verify final layout matches recipe intent

**Recipe: dev (terminal 60% left, browser 40% right)**
```bash
i3-msg 'workspace 1; layout splith'
i3-msg '[class="Alacritty"] focus; resize set 60 ppt 0 ppt'
i3-msg '[class="Brave-browser"] move container to workspace number 1'
```

**Recipe: pair (two terminals 50/50)**
```bash
i3-msg 'workspace 1; layout splith'
# Identify two terminal windows and arrange
```

### 4. Screenshot & Observe (Layer 1: "See")
Claude is multimodal — screenshots enable visual awareness of the desktop.

**Capture patterns:**
```bash
# Full screen capture
flameshot full -p /tmp/claude-i3-screenshot.png

# Capture specific window by focusing it first, then using its geometry
WID=$(xdotool search --class "Brave-browser" | head -1)
GEOM=$(xdotool getwindowgeometry --shell "$WID")  # outputs X, Y, WIDTH, HEIGHT
# Use flameshot for region: flameshot full -p /tmp/ then crop, or import -window
```

**Observe pattern (screenshot → analyze → suggest):**
1. Capture screenshot to /tmp/claude-i3-screenshot.png
2. Read the image file with the Read tool (multimodal)
3. Describe what's visible and suggest actions
4. Clean up the screenshot file when done

**IMPORTANT:** Always `rm /tmp/claude-i3-screenshot.png` after reading it.

### 5. Keyboard Simulation (Layer 2: "Type")
Use `xdotool` to send keystrokes to the focused window.

**Safety protocol — ALWAYS follow this order:**
1. Confirm which window is focused (`i3-msg -t get_tree | jq focused`)
2. If targeting a specific window, focus it first via i3-msg criteria
3. Brief sleep after focus switch (`sleep 0.1`) to let X11 settle
4. Then send keystrokes

**Typing text:**
```bash
# Type text with inter-key delay (ms) to avoid dropped chars
xdotool type --delay 12 "hello world"

# Type into a specific window (focus first via i3, more reliable than xdotool search)
i3-msg '[class="Brave-browser"] focus' && sleep 0.1 && xdotool type --delay 12 "search query"

# For special characters or passwords, use xdotool with --clearmodifiers
xdotool type --clearmodifiers --delay 12 "text with @special #chars"
```

**Key combinations:**
```bash
# Single keys
xdotool key Return
xdotool key Escape
xdotool key Tab
xdotool key BackSpace

# Modifier combos (use lowercase modifier names)
xdotool key ctrl+s           # save
xdotool key ctrl+c           # copy
xdotool key ctrl+v           # paste
xdotool key ctrl+shift+t     # reopen tab in browser
xdotool key alt+F4           # close window
xdotool key ctrl+l           # browser address bar
xdotool key super+l          # (if bound to lock)

# Multiple keys in sequence
xdotool key ctrl+a && sleep 0.05 && xdotool key ctrl+c  # select all + copy
```

**IMPORTANT:** `xdotool type` sends raw characters. For commands that need modifier keys
(Ctrl+C is NOT the letter C with ctrl), always use `xdotool key`, not `xdotool type`.

### 6. Mouse Simulation (Layer 3: "Click")
Use `xdotool` for mouse operations. Coordinate-based — less reliable than keyboard,
use only when no better alternative exists (prefer i3 IPC, Playwright for browsers).

**Safety protocol for clicks:**
1. Screenshot first to understand what's at the target coordinates
2. Confirm the target visually before clicking
3. Screenshot after to verify the result
4. For important clicks, describe to the user what you're about to click and why

**Click patterns:**
```bash
# Get current mouse position (useful for debugging)
xdotool getmouselocation

# Left-click at absolute screen coordinates
xdotool mousemove 500 300 click 1

# Right-click
xdotool mousemove 500 300 click 3

# Middle-click
xdotool mousemove 500 300 click 2

# Double-click
xdotool mousemove 500 300 click --repeat 2 --delay 50 1

# Click relative to a window (more stable than absolute coords)
WID=$(xdotool search --class "Brave-browser" | head -1)
xdotool mousemove --window "$WID" 100 50 click 1

# Click then return mouse to original position
ORIG=$(xdotool getmouselocation --shell)
xdotool mousemove 500 300 click 1
eval "$ORIG" && xdotool mousemove "$X" "$Y"
```

**Scroll patterns:**
```bash
# Scroll up (button 4) / down (button 5)
xdotool click --repeat 3 --delay 50 5   # scroll down 3 clicks
xdotool click --repeat 3 --delay 50 4   # scroll up 3 clicks

# Scroll at specific position
xdotool mousemove 500 300 click --repeat 5 --delay 50 5
```

**Drag patterns:**
```bash
# Click-drag from (x1,y1) to (x2,y2)
xdotool mousemove 100 200 mousedown 1 mousemove 500 400 mouseup 1
```

### 7. Clipboard Operations
`xclip` binary is at `xclip`.
Define alias for convenience.

```bash
XCLIP="xclip"

# Write to clipboard
echo -n "text to copy" | "$XCLIP" -selection clipboard

# Read from clipboard
"$XCLIP" -selection clipboard -o

# Copy focused window's selected text (simulate Ctrl+C then read clipboard)
xdotool key ctrl+c && sleep 0.1 && "$XCLIP" -selection clipboard -o

# Paste text (write to clipboard then simulate Ctrl+V)
echo -n "text to paste" | "$XCLIP" -selection clipboard && sleep 0.05 && xdotool key ctrl+v
```

**IMPORTANT:** Reading clipboard may expose sensitive data (passwords, tokens).
Only read clipboard when the user explicitly requests it. Never log clipboard contents
to files or include in commit messages.

### 8. Desktop Notifications
Use `notify-send` for dunst integration:
```bash
notify-send "Title" "Message body"
notify-send -u critical "Alert" "Something important"
notify-send -t 5000 "Timed" "Disappears in 5 seconds"
```

### 9. Compound Interaction Patterns
These combine multiple layers for real workflows.

**Pattern: "Read text from a GUI window"**
```bash
# 1. Focus the window
i3-msg '[class="Brave-browser"] focus' && sleep 0.1
# 2. Select all + copy
xdotool key ctrl+a && sleep 0.05 && xdotool key ctrl+c && sleep 0.1
# 3. Read clipboard
XCLIP="xclip"
"$XCLIP" -selection clipboard -o
```

**Pattern: "Fill a text field in a GUI app"**
```bash
# 1. Focus the app
i3-msg '[class="TargetApp"] focus' && sleep 0.1
# 2. Click the field (if needed — get coords from screenshot)
xdotool mousemove 400 300 click 1 && sleep 0.1
# 3. Clear existing text and type new
xdotool key ctrl+a && sleep 0.05 && xdotool type --delay 12 "new text"
```

**Pattern: "Navigate browser to URL"**
```bash
# 1. Focus browser
i3-msg '[class="Brave-browser"] focus' && sleep 0.1
# 2. Open address bar
xdotool key ctrl+l && sleep 0.1
# 3. Type URL and go
xdotool type --delay 8 "https://example.com" && sleep 0.05 && xdotool key Return
```

**Pattern: "Screenshot-verify-act loop"**
```bash
# 1. Screenshot
flameshot full -p /tmp/claude-i3-screenshot.png
# 2. Read image (multimodal) — analyze what's visible
# 3. Decide action based on what's seen
# 4. Execute action (click, type, etc.)
# 5. Screenshot again to verify
flameshot full -p /tmp/claude-i3-screenshot.png
# 6. Read and confirm result
# 7. Clean up
rm -f /tmp/claude-i3-screenshot.png
```

**IMPORTANT:** For browser automation beyond simple URL navigation,
prefer Playwright MCP tools — they're more reliable than coordinate-based clicking.
Reserve xdotool mouse interaction for native GUI apps that have no programmatic API.

## IPC Reference

### Query Commands (via `i3-msg -t <type>`)
| Type | Returns |
|------|---------|
| get_tree | Full container tree with all windows, layouts, geometry |
| get_workspaces | Workspace list: num, name, visible, focused, urgent, output, rect |
| get_outputs | Monitor list: name, active, primary, current_workspace, rect |
| get_marks | Array of all mark names |
| get_binding_modes | Available modes (default, resize) |
| get_bar_config | Bar IDs; with ID arg returns full bar config |
| get_version | i3 version info |
| get_config | Loaded config file contents |
| get_binding_state | Current binding mode name |

### Container Tree Node Fields
Key fields when parsing `get_tree`:
- `type`: "root", "output", "con", "floating_con", "workspace", "dockarea"
- `name`: window title or workspace name
- `window`: X11 window ID (null for containers)
- `window_properties`: {class, instance, title, window_role, machine}
- `focused`: boolean
- `urgent`: boolean
- `layout`: "splith", "splitv", "stacked", "tabbed"
- `marks`: array of mark strings
- `rect`: {x, y, width, height}
- `nodes`: tiling children
- `floating_nodes`: floating children
- `fullscreen_mode`: 0 (none), 1 (output), 2 (global)
- `floating`: "auto_on", "auto_off", "user_on", "user_off"

### Action Commands (via `i3-msg '<command>'`)
- **Focus**: `focus left|right|up|down|parent|child|floating|tiling|mode_toggle`
- **Move**: `move left|right|up|down|to workspace <n>|to output <name>`
- **Workspace**: `workspace <n>|next|prev|back_and_forth`
- **Layout**: `layout splith|splitv|tabbed|stacking|toggle [layouts...]`
- **Split**: `split h|v|toggle`
- **Floating**: `floating enable|disable|toggle`
- **Fullscreen**: `fullscreen enable|disable|toggle`
- **Sticky**: `sticky enable|disable|toggle`
- **Border**: `border normal|pixel|none`
- **Resize**: `resize grow|shrink width|height <n> px [or <n> ppt]`
- **Resize set**: `resize set <width> ppt <height> ppt`
- **Mark**: `mark [--add|--replace|--toggle] <name>`
- **Unmark**: `unmark [name]`
- **Scratchpad**: `move scratchpad`, `scratchpad show`
- **Exec**: `exec [--no-startup-id] <command>`
- **Rename**: `rename workspace <old> to <new>`
- **Reload/Restart**: `reload`, `restart`
- **Exit**: `exit` (REQUIRES explicit user confirmation)

### Criteria Selectors
Prefix commands with `[criteria]` to target specific windows:
```
[class="X"]        Window class (WM_CLASS second value)
[instance="X"]     Window instance (WM_CLASS first value)
[title="X"]        Window title (supports regex)
[con_mark="X"]     Container mark
[workspace="X"]    Windows on workspace
[con_id=N]         Container ID (from get_tree)
[id=N]             X11 window ID
[urgent=latest]    Most recently urgent window
[floating]          Floating windows only
[tiling]            Tiling windows only
[window_role="X"]  Window role
```

Multiple criteria combine as AND: `[class="Alacritty" title=".*vim.*"]`

## Reliability Tiers
Choose the most reliable method available for each task:

**Tier 1 — Deterministic (prefer these):**
- i3 IPC via `i3-msg`: window focus, move, layout, workspace, marks, criteria selectors
- `xdotool type`: text input (deterministic character sequence)
- `xdotool key`: key combos (Ctrl+S, Return, Escape — deterministic)
- Clipboard via `xclip`: data transfer in/out of GUI apps
- `notify-send`: desktop notifications

**Tier 2 — Visual verification recommended:**
- `xdotool mousemove + click`: coordinate-based clicking (fragile if window moves/resizes)
- `xdotool mousedown/mouseup`: drag operations
- Scroll operations: may need adjustment for scroll speed/amount
- Arrange recipes: compound operations that depend on window state

**Tier 3 — Use only as last resort:**
- Multi-step GUI automation (click → type → click → verify) — each step compounds error risk
- Coordinate-based operations on dynamic content (scrolling pages, animations)
- Browser GUI interaction — prefer Playwright MCP tools instead

**Rule:** Always use the highest-tier method available. If Tier 2/3 is needed, bracket
the action with screenshots (before + after) to verify the result.

## Boundaries

**Will:**
- Query and display i3 state in readable formats
- Execute i3 commands to manage windows, workspaces, layouts
- Apply layout recipes for common arrangements
- Use criteria selectors for precise window targeting
- Send notifications via dunst (notify-send)
- Mark windows for quick navigation
- Capture and analyze screenshots (multimodal vision)
- Type text and send key combos to focused windows via xdotool
- Click at coordinates when no better method exists (with screenshot verification)
- Read/write the X11 clipboard for data transfer with GUI apps
- Combine layers for compound workflows (focus → screenshot → type → verify)

**Will Not:**
- Modify i3 config files (use /devrc-dx for that)
- Reload or restart i3 without explicit user request
- Exit i3 — always requires explicit confirmation, warn before executing
- Override keybindings at runtime
- Execute destructive operations on windows without stating what will happen first
- Read clipboard without explicit user request (may contain secrets)
- Send keystrokes without first confirming which window is focused
- Chain more than 3 blind actions without a screenshot verification step
- Use xdotool mouse for browser automation when Playwright is available
- Leave screenshot files in /tmp after reading them
