# devrc — NixOS / home-manager dotfiles

Personal dev-environment config (zsh, tmux, neovim, i3, scripts) for the workbench + laptop NixOS hosts. Managed by **home-manager via a flake**.

## Shell environment (read before running commands)
- **Bash tool runs NON-interactive zsh** (`zsh -c`) → sources `.zshenv` only, NOT `.zshrc`/`initContent`. Shell tweaks Claude needs at runtime must go in home-manager `programs.zsh.envExtra` (→ `.zshenv`). `unsetopt nomatch` lives there so unmatched globs pass through literally instead of aborting with "no matches found".
- zsh reserves `status` — use `rc=`/`out=`, never `status=$(...)`.
- Use `git -C <path>` and absolute paths — never `cd <repo> && …` (triggers approval prompts and can run untrusted hooks).

## Applying changes
- **Deploy to BOTH hosts (after merge):** `scripts/ship.sh` — converges workbench + laptop to `origin/main` (stash → ff-pull → `home-manager switch` → pop → verify HEAD==origin/main) in one idempotent call. Use this instead of hand-running the per-host dance; `--no-laptop`/`--no-local` to scope. Covers home-manager only (not `sudo nixos-rebuild`).
- **Apply config (single host):** `home-manager switch --flake ~/workspace/devrc --impure` (allowlisted). This is how you validate a Nix edit end-to-end.
- **Quick syntax check** before switching: `nix-instantiate --parse <file>.nix >/dev/null`.
- **NEVER `sudo nixos-rebuild` from Claude** — can't sudo non-interactively. System-level changes must be staged as an apply script for the user to run (see the `laptop` skill's `stage-system` pattern). home-manager (user-level) is fine.

## Server / headless mode
- `~/.server-mode` marker toggles graphical bits: `headless-mode` (disables dunst/espanso) vs `graphical-mode` (re-enables), both run a home-manager switch. A host may be in server mode — check for the marker before assuming a GUI.

## Layout
- `nix/` — home-manager modules (`programs/zsh`, tmux, nvim, i3, …). `flake.nix` at root.
- `scripts/` — utility scripts (prefer extending these over re-typing inline bash / heredocs).
- `scripts/collector/` + `scripts/validation/` — the **personal activity-telemetry pipeline**: 6 sources (zsh/tmux hooks, X11 keylogger, Brave ext [nav+scroll], Claude-session tailer, i3 window/workspace focus) → `emit` → `collector.py` (systemd user daemon) → dedicated authed homelab ClickHouse `activity.events` → Grafana dashboard; `validation/` is the deterministic harness; `scripts/session-analysis/activity-scan.py` is the on-demand "where time goes + what to automate" report; `scripts/session-analysis/initiative-scan.py` is the **cross-repo initiative + progress ledger** (handoff docs + git + telemetry → momentum/next-step), operated via the **`/initiatives`** slash command. **Operate it via the `activity` skill** (endpoints, creds, gotchas, deploy). Gotchas that bite: `ts` is UTC (dashboard buckets local); laptop must use its nebula CH endpoint; both hosts are hostname `nixos` so `ACTIVITY_HOST` disambiguates; `home-manager switch` DOES restart the daemons on a script change now (X-Restart-Triggers). Browser `active_ms` was retired (broken on i3) → attention is i3-derived.
- `scripts/mail-actions/` — **email-automation layer** over the self-hosted mail inbox (the Postgres `mail` table; separate from activity telemetry). Deterministic Stage-1 filter → LLM action-required extraction → `mail_actions` queue (one live action per thread, auto-closes when you reply) + invoice→`taxes-{year}-invoices` MinIO archiver (daily systemd timer `mail-actions-archive`) + a sibling Gmail Sent-mail IMAP poller running as a homelab CronJob. Action-extractor is manual (`extract.py run`, the only LLM/$ piece); archiver is scheduled. **Operate via the `mailbox` skill.**
- `scripts/repo-cos/` — **repo "chief-of-staff"** ("agents bring me ideas" / CEO model): weekly `serverMode`-gated workbench timer (`repo-cos.timer`, Mon 08:00) deterministically scans the repos for improvement signals (TODO/FIXME, skipped tests, `latest` tags, churn) → cheap LLM synthesis (OpenRouter, survivors only) → top-5 evidence-backed proposal digest. **SELF-HOSTED mail** (defaults): **sends** via Zach's postfix relay `From: repo-cos@mail.zacx.dev` (DKIM-signed; via a `kubectl port-forward` to the production cluster), **Reply-To `repo-cos@inbox.zacx.dev`**; his reply routes to the homelab Postgres `mail` table and `feedback.py` **reads it from Postgres** (via `mail-actions/_db.py`). Gmail-SMTP/IMAP kept as fallbacks (`REPO_COS_SEND`/`REPO_COS_REPLY_SRC`). `exclusions.py` then deterministically acts on the reply TWO ways: **repo-level** ("paused"/"not owner" → HARD-DROP the repo; `resume <repo>`) and **recommendation-level** ("skip"/"not needed" → dismiss just that proposal by its evidence `file:line`, keep the repo in scope) — both in `~/.config/repo-cos/exclusions.json` (`"repos"`+`"dismissed"`), + context-injects the rest. Needs BOTH kubeconfigs (prod relay + homelab pg) + `kubectl`/`psycopg2` (best-effort). Creds: `~/.config/repo-cos/env` (OpenRouter). **Operate via the `repo-cos` skill.** ⚠ gotcha: `feedback.py` loads `mail-actions/_db.py` by explicit importlib path — do NOT add `mail-actions/` to `sys.path` (its `llm.py` shadows repo-cos's → breaks synthesis).
- `claude/` — **global Claude Code config managed declaratively** (`RULES.md`, `PRINCIPLES.md`, and **all slash-commands under `claude/commands/`**). `nix/home.nix` symlinks these to `~/.claude/` via `home.file` (commands recursively → each lands as `~/.claude/commands/<name>.md`), so both hosts stay in sync (they were drifting when edited per-host). **Edit `claude/*.md` / `claude/commands/*.md` here + `home-manager switch`/`ship.sh` — NOT `~/.claude/*` (read-only nix-store symlinks).** Flake gotcha: a NEW command file must be `git add`ed before it switches (flakes only see tracked files). New-host caveat: `home.file.force` does NOT clobber a pre-existing *foreign* `~/.claude/RULES.md` or `~/.claude/commands/*.md`; `rm` the foreign ones once before the first switch. (`~/.claude/CLAUDE.md` and `skills/` stay per-host — host-specific + frequently edited in place.)
- `.zshrc`, `.tmux.conf` etc. are read by the nix modules — read with offset/limit, they're large.

## Conventions
- Never `git add -A` — stage files individually.
- Commit + push so both hosts (workbench + laptop) can `git pull` then `home-manager switch`.
